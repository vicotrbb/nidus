# Product

## Register

brand

## Users

Nidus serves senior Rust backend engineers and framework evaluators who want a
modular service framework without hiding the Rust ecosystem. They use the site
to evaluate installation, API shape, examples, production defaults, benchmark
evidence, and whether the framework is credible for a real service.

## Product Purpose

The website is the public evaluation and documentation surface for Nidus. It
should help an engineer understand the framework quickly, verify its explicit
composition model and adapter boundaries, reproduce its evidence, and move from
evaluation to a working application without marketing claims outrunning proof.

## Brand Personality

Sharp, deliberate, inspectable. The interface should feel technically rigorous
and public-release ready, with a light framework-manual surface and dark ink text
rather than a generic developer-tool theme. The intended emotional outcome is
calm confidence grounded in visible technical evidence.

## Anti-references

Avoid full-page black or green developer-tool themes, generic AI gradients,
glassy glow-heavy surfaces, decorative visuals that do not explain the product,
and marketing-only layouts that bury installation, documentation, examples, or
proof. Do not imitate another framework's visual identity or make universal
performance claims from bounded benchmarks.

## Design Principles

- Make installation, documentation navigation, and real examples immediately visible.
- Keep the core facade and official adapters distinct so the lean-core story is obvious.
- Use the logo-derived organic nesting motif as the memorable visual element.
- Prefer dense, scannable technical evidence over marketing-only composition.
- Keep claims bounded, reproducible, and visibly connected to their methodology.
- Keep the site light-first, reserving dark surfaces for code and focused technical panels.

## Accessibility & Inclusion

Target WCAG 2.2 AA for contrast, keyboard navigation, focus visibility, and
semantic structure. Preserve readable code and table content on narrow screens,
support reduced-motion preferences, avoid color-only meaning, and verify both
desktop and mobile layouts.
