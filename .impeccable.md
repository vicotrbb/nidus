## Design Context

### Users

Nidus serves senior Rust backend engineers and framework evaluators who want a modular service framework without hiding the Rust ecosystem. They are likely reading the site while evaluating installability, API shape, examples, production defaults, and whether the framework is stable enough for a real service.

### Brand Personality

Sharp, deliberate, inspectable. The interface should feel technically rigorous and public-release ready, with a light framework-manual surface and dark ink text rather than a generic developer-tool theme.

### Aesthetic Direction

Use a light pearl and violet-tinted structure with restrained violet identity accents, dark code panels, sharp organic nesting forms inspired by the Nidus logo, and documentation-first information architecture. The site may take structural inspiration from documentation-forward framework sites with immediate install commands, code panels, and evaluator paths, but it must present Nidus as its own Rust framework. Avoid full-page black backgrounds, green backgrounds, generic AI gradients, glassy glow-heavy surfaces, and decorative visuals that do not explain the product.

### Design Principles

- Make installation, docs navigation, and real examples immediately visible.
- Keep the core facade and official adapters visually distinct so the lean-core story is obvious.
- Use the logo-derived organic nesting motif as the memorable visual element.
- Prefer dense, scannable documentation layouts over marketing-only composition.
- Keep the site light-first, reserving dark surfaces for code and focused technical panels.
- Verify desktop and mobile layouts so code samples, nav, and calls to action remain readable.
